# Patronage

A cozy city builder set in Renaissance Italy. You play the patron: build the city that attracts painters and sculptors, keep them supplied and inspired, and commission the masterworks that bring on the Renaissance.

![A coastal city in Patronage — cathedral, plazas, and terracotta rooftops beside the sea](docs/screenshot-hero.png)

## The game

Every artwork in Patronage is **commissioned** — the Church, noble families, and guilds offer contracts with a name, a patron, and a deadline. You accept a commission, assign it to a workshop, and the artists housed there work it month by month — as long as they have materials, staff, and a city worth being inspired by.

The economy is deliberately lean — three headline resources and a population pool:

- **Florins** — what you spend. Earned by markets; spent on buildings.
- **Inspiration** — what you cultivate. Generated by plazas, decorations, and displayed masterworks; fuels artist arrivals and progress.
- **Prestige** — what you earn. Completed commissions build your city's reputation toward the Renaissance milestone.
- **Population** — the labor pool that staffs everything, capped by housing and services.

A few design rules shape everything else: no citizen pathfinding or supply-chain micromanagement (service buildings are threshold checks, materials are supplier *capacity*, never a stockpile); spatial bonuses are soft nudges with gentle falloff, never hard cutoffs; and every building either unlocks something or passively boosts something — nothing demands babysitting. The full spec lives in [docs/design-doc.md](docs/design-doc.md).

The scarcity that drives decisions: supplier capacity. A Pigment Trader serves a limited number of painters; when demand outgrows capacity, somebody's fresco waits. Which artists get materials is the game's recurring question.

![Managing commissions and the artist roster](docs/screenshot-ui.png)

**Status:** playable and in active development. Core systems are in — placement, the tick loop, worker allocation, artists with a seven-rank career ladder, supplier gating, the commission pipeline, plaza connectivity, and procedurally generated water maps. The roadmap (masterwork display, teaching, the Renaissance celebration itself) is tracked in the design doc's development phases.

## How it's built

Two layers share one screen: a Babylon.js canvas renders the 3D city, and a React DOM overlay renders every panel and tooltip on top of it. A Zustand store is the seam between them.

```
app/game/         pure simulation — building defs, tick loop, worker allocation,
                  commissions, connectivity, procedural water. No rendering imports.
app/game/render/  Babylon.js — terrain, water, models, batching, placement ghosts
app/game/ui/      React overlay — panels, tooltips, build palette
app/stores/       Zustand store wiring sim to UI
```

Things under the hood that were fun to build:

- **A pure, tested simulation core.** The sim layer has no rendering dependencies, so the whole game logic runs headlessly under `node:test` (13 check suites — `npm test`). Tick transitions are pure functions, which makes integration coverage cheap.
- **Deterministic, seeded world generation.** A short shareable run seed drives everything procedural: the city's name, the map archetype (five, from dry plains to a coastal estuary), the river's course, terrain hills, and wilderness scatter. Same seed, same world — the water layer derives entirely from the seed at load time and is never stored.
- **Two-pass worker allocation.** Workers first fill every building to minimum staffing in priority order, then spread up to max capacity for efficiency bonuses — so a labor shortage degrades the city gracefully instead of randomly.
- **Soft spatial bonuses via 0-1 BFS.** The Main Plaza's bonus radiates through the road network with linear falloff, refreshed by secondary plazas. Connection is a carrot (up to +25%), never a requirement — off-network buildings run at full base rate.
- **Versioned save migrations that preserve old cities.** Example: when water maps landed, existing saves got `mapSeed: null` — forever dry — because rolling a new river under a built city would flood someone's cathedral.
- **Rendering that scales with city size.** Placed buildings render as thin-instance batches (one draw call per source mesh, not per building), shadows refresh on demand, and the terrain/water meshes are generated flat-shaded geometry with a CPU vertex wobble for the water surface.
- **Kitbashed art pipeline.** Buildings are composed from CC0 Kenney kit pieces in a declarative prefab manifest, retinted to a Mediterranean palette by patching the kits' shared colormap ([scripts/retint-colormap.py](scripts/retint-colormap.py)).

**Stack:** TypeScript, React 19, React Router v7, Babylon.js, Zustand, Tailwind CSS v4, Vite.

## Running it

```bash
npm install
npm run dev        # http://localhost:5173
npm test           # sim check suites (node:test)
npm run typecheck  # strict TS
```

Or with Docker:

```bash
docker build -t patronage .
docker run -p 3000:3000 patronage
```

Dev flags (dev server only): `/?demo` loads a seeded showcase city, `&pause` freezes time, `&map=<seed>` forces a specific map's water layer, `&cam=x,z[,radius]` frames the camera for screenshots.
